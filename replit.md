# Probaly - AI Sports Prediction App

## Overview
Probaly is a mobile sports prediction application available on iOS and Android, developed using React Native and Expo. It leverages AI to provide probability-based insights for sports events. The app operates on a freemium model, offering a daily free tip and a premium subscription priced at $49/year. The business vision is to become a leading platform for AI-driven sports analytics, offering users a competitive edge in sports predictions.

## User Preferences
- **Production-First**: All development work should target the production server (probaly.net). The development server (localhost:5000) is only used for testing code before deploying. Every code change must be deployed to production to take effect. No dev-only workarounds.
- The agent should prioritize security best practices in all development tasks. When implementing new features, consider the impact on app store compliance and user privacy. The agent should also ensure that the application's performance remains optimal, especially concerning API response times and prediction generation. Do not make changes to the folder `shared`.

## System Architecture

### UI/UX Decisions
The application features a 5-tab navigation system (Home, Live, Sports, History, Profile). The brand utilizes a color scheme of Navy Blue (`#1A237E`) as primary and Bold Red (`#E53935`) as accent. Success indicators are Emerald (`#10B981`) and warnings are Amber (`#F59E0B`).

### Technical Implementations
- **Frontend**: React Native with Expo for cross-platform mobile development.
- **Backend**: Express.js in TypeScript.
- **Database**: PostgreSQL with Drizzle ORM.
- **Authentication**: Email/password with bcrypt for hashing and JWT for session management (30-day expiry). Registration verifies the email's domain is deliverable via DNS (MX with A/AAAA fallback per RFC 5321) before creating an account; this check runs before the duplicate-email check to keep response timing similar between existing and new emails. Validator enforces RFC 7505 Null MX (rejects domains with single empty-exchange MX, no A-record fallback), filters bogus MX hosts (`0.0.0.0`, `localhost`, etc.), and ships a built-in disposable-domain blocklist (mailinator, tempmail, guerrillamail, yopmail, plus known-abused custom domains like qowo.com/gausi.com/jui.com/gma.com). Fails open on transient DNS errors. Implemented in `server/emailValidation.ts`.
- **AI Integration**: Groq (GROQ_API_KEY, model: llama-3.3-70b-versatile) powers all prediction generation and batch resolution — free tier, OpenAI-compatible. OpenAI is not used.
- **Payment Processing**: RevenueCat handles in-app purchases for iOS and Android. Stripe is used exclusively for affiliate payouts.
- **Server-Side RC Verification**: `server/revenueCatService.ts` provides `checkRCSubscription(userId)` which calls RevenueCat's v1 REST API using the Replit-managed OAuth token. Called by `GET /api/subscription/:userId` (checks RC when user is not premium in DB) and `POST /api/revenuecat/sync` (verifies native purchases server-side). Stripe web payments bypass RC check (identified by `stripe_*` productIdentifier). Falls back gracefully when RC API is unavailable.
- **Data Refresh**: A daily scheduler refreshes predictions, clears expired entries, fetches new games, and regenerates AI predictions. After refresh, push notifications are sent to all registered devices to alert users that the daily free tip is ready.
- **System-Pick Generation Locking** (`server/services/predictionService.ts`): A single shared async mutex `systemPickLock` (acquired via `acquireSystemPickLock()`) serializes ALL system-pick (`userId IS NULL`) generation. `_generateDailyFreeTip` and `generateDemoPredictions` each grab the lock at entry and release in `finally`. This eliminates two race conditions that previously produced duplicate system predictions: (1) `resetAndGenerateDailyFreeTip` (midnight) racing with the 30-min `checkAndReplaceFreeTip` tick on the same free tip, and (2) `_generateDailyFreeTip` inserting AFTER `generateDemoPredictions` snapshotted `existingSystemPicks` so the demo pool generated a premium pick for the same game. Belt-and-braces: both functions also do a last-second `(matchTitle, date(matchTime), userId IS NULL)` collision check directly against the DB right before each insert — guarantees no duplicate even if a future call site bypasses the mutex. The legacy `isGeneratingFreeTip` flag is left in place but the lock is the source of truth.
- **Push Notifications**: Uses Expo Push Notifications API. Client registers push tokens on login/signup/app restart. Server stores tokens in `push_tokens` table and sends via Expo's push API. Respects user notification preferences.
- **History Filtering Rules (NEVER change without explicit instruction)**:
  - **Premium users**: See only `correct` predictions — both premium picks and free daily tips — that were real pre-game predictions (`expiresAt > matchTime`), within last 30 days. Retroactive ESPN entries (`expiresAt = matchTime`) never shown. Incorrect predictions are never shown.
  - **Free users**: Correct picks only, split into two windows: real free daily tips (`isPremium = false`, `expiresAt > matchTime`) shown for 30 days; retroactive ESPN history entries (`expiresAt = matchTime`) shown for 5 days. Incorrect predictions are NEVER shown.
- **Affiliate Program**: Currently disabled. Code retained in `server/affiliateRoutes.ts` and `client/screens/AffiliateScreen.tsx` for future re-enablement. Was: referral code generation, commission tracking (40%), 14-day clearance, manual payout approval via Stripe Connect.
- **Security**: Implements real JWT authentication, role-based access control (`requireAuth`, `requireAdmin`), per-route rate limiting, security headers (HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy), tightened CORS policies, RevenueCat webhook authorization (via `REVENUECAT_WEBHOOK_SECRET`), and Stripe webhook signature verification.
- **Deployment**: Production server binds to port 8081 (set via deployment run command `PORT=8081 npm run server:prod`) to match Replit's port mapping (external 80 → local 8081). Dev server uses port 5000.
- **App Store Compliance**: Includes iOS Privacy Manifest, deep linking (`probaly://`), a gambling disclaimer, and legal page accessibility.

### Feature Specifications
- **AI-Powered Predictions**: Each prediction includes probability, confidence, explanation, factors, and a risk index. Basketball games include over/under total points predictions alongside winner predictions.
- **Over/Under Bets (Basketball)**: AI generates game total (over/under) predictions for basketball matches with realistic point lines. Match titles use `(O/U)` suffix. Result resolution compares total score against the line. History entries include O/U records for completed basketball games.
- **Live Match Updates**: Provides real-time updates for ongoing games.
- **Prediction History**: Users can track their past predictions.
- **Subscription Model**: Offers a "Probaly Premium" subscription with monthly ($49.99) and annual ($149.00) options. iOS/Android uses RevenueCat for in-app purchases. Web browser users use Stripe Checkout (redirects to Stripe-hosted payment page, webhook activates premium).
- **Sports Data Fallback**: Automatically switches to ESPN's free public API if The Odds API quota is exhausted, ensuring continuous data availability.
- **Telegram Channel Mirror (Landing Page)**: `server/services/telegramService.ts` connects to the private Telegram channel `t.me/+5uZNUktfpeZiMjVi` via gramjs MTProto using a personal user-account session (secrets `TELEGRAM_API_ID`, `TELEGRAM_API_HASH`, `TELEGRAM_SESSION_STRING` — generate via `npx tsx scripts/telegramLogin.ts`). Listens for new photos/videos, downloads to `server/uploads/telegram/`, stores metadata in the `telegram_media` table (raw-SQL bootstrapped on startup, NOT in shared/ schema; an idempotent `ALTER TABLE ... ADD COLUMN IF NOT EXISTS activated_at` runs on every boot). On startup, backfills the last 30 channel messages (skipping any already past the 24h window). The landing page (`/`) shows up to 3 items via `GET /api/landing/telegram-media`, polled every 60s. **Reconnect backoff**: `pollForNewMedia` tracks `consecutiveConnectFailures` and uses an exponential backoff schedule (`backoffDelayForFailures`): failures 1-2 retry on the normal 90s poll cadence; failure 3 waits 5 min; failure 4 waits 15 min; failure 5+ waits 30 min (capped). The counter resets to 0 on the first successful reconnect. This prevents a session-flag feedback loop — Telegram only ever sees a slow trickle of connect requests when something is wrong, never a storm. **Display semantics**: the landing page always shows the latest 3 posts by `created_at` (no `activated_at` filter), so a newly-ingested photo/video is visible within seconds of arriving in the channel. Cleanup never deletes the latest 3 (even if their `expires_at` has elapsed) to guarantee the gallery is never empty during a Telegram outage or quiet period. The rotation flow below still runs to refresh `expires_at` on the latest 3 (so cleanup doesn't churn through them) and to drive the daily 11 AM ET tick. **Display rotation**: every newly ingested live post triggers an immediate rotation — `ingestMessage` now returns a boolean indicating whether a new row was inserted, and the `NewMessage` handler calls `rotateDailyDisplay()` the moment it gets `true`, so a brand-new photo/video appears on the landing page within seconds (the oldest of the previously-displayed 3 is naturally pushed out of the top `MAX_DISPLAY_ITEMS` by `getActiveMedia`'s `ORDER BY activated_at DESC`). The 11 AM America/New_York scheduled rotation is retained as a safety net for quiet days: a 60s polling tick (`checkRotation`) runs `rotateDailyDisplay()` the first time it sees ET hour ≥ 11 each ET day. The instant-rotation path also marks `lastRotationDateET = today` (when already past 11 AM ET) so the scheduled tick doesn't run a redundant rotation later that day. Backfill does NOT trigger per-message rotation — the existing post-backfill `ensureInitialRotation()` handles that case once. On startup, `ensureInitialRotation()` does a one-off rotation if no rows are currently active (so a fresh deploy or post-downtime restart never shows an empty gallery). Cleanup (10 min interval): non-activated rows expire 24h after their post time; activated rows stay alive until the next rotation deactivates them by being the next "3 newest". The SPA catch-all in `configureExpoAndLanding` exempts `/uploads/` so static media serves correctly. Disabled gracefully if any of the three secrets are missing.

## External Dependencies
- **OpenAI**: Used for AI-powered sports predictions (GPT-4o).
- **RevenueCat**: Manages in-app subscriptions and purchases on iOS and Android.
- **Stripe**: Handles web browser premium subscriptions via Stripe Checkout. Price IDs are configured via `EXPO_PUBLIC_STRIPE_PRICE_MONTHLY` and `EXPO_PUBLIC_STRIPE_PRICE_ANNUAL` environment variables (shared). The server exposes `/api/billing/config` as the canonical source; the client fetches from there with env var fallback. Affiliate payout via Stripe Connect is currently disabled.
- **The Odds API**: Primary source for real-time sports event data.
- **ESPN API**: Secondary data source for sports events, used as a fallback.
